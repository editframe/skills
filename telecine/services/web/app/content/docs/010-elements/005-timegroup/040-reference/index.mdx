### Grouped Attributes

#### Temporal Composition

<PropertyDocList>
  <PropertyDoc
    name="mode"
    domReadable
    domWritable
    htmlAttribute
    type="string"
    defaultValue="contain"
  >
    The temporal mode of the timegroup. Can be `fixed`, `sequence`, `contain`, or `fit`.

    - **`fixed`**: Uses explicit duration attribute (base case)
    - **`sequence`**: Sum of child durations minus overlaps - children play sequentially
    - **`contain`**: Maximum of child durations - children play simultaneously (parallel)
    - **`fit`**: Inherits duration from parent timegroup - perfect for backgrounds

    For detailed explanation of how modes work, see [Temporal Composition](/docs/elements/timegroup/explanation/temporal-composition).
  </PropertyDoc>
  <PropertyDoc
    name="overlap"
    domReadable
    domWritable
    htmlAttribute
    type="timestring"
    defaultValue="0s"
  >
    The overlap duration between sequential items in a sequence timegroup. This property is only applied when the `mode` attribute is set to `sequence`.

    Overlap creates **temporal intersection**, not visual crossfade. Visual transitions require explicit CSS animations using `--ef-transition-duration` and `--ef-transition-out-start` variables.

    This can be combined with CSS animation to create transitions between sequence items. See [How to Use CSS Variables](/docs/elements/timegroup/how-to/css-variables) for examples.
  </PropertyDoc>
  <PropertyDoc
    name="duration"
    domReadable
    domWritable
    htmlAttribute
    type="timestring"
    defaultValue=""
  >
    The duration of the timegroup. Values written to this property are only applied when the timegroup is in `fixed` mode.

    Otherwise the value is computed based on the duration of the children:

    - **Sequence mode**: Sum of child durations minus overlaps
    - **Contain mode**: Maximum of child durations
    - **Fit mode**: Inherits from parent timegroup

    For details on duration calculation, see [Temporal Composition](/docs/elements/timegroup/explanation/temporal-composition).
  </PropertyDoc>
</PropertyDocList>

#### Time Coordinates

<PropertyDocList>
  <PropertyDoc
    name="currentTime"
    domReadable
    domWritable
    htmlAttribute="currenttime"
    type="number"
    defaultValue="0"
  >
    **Type**: `number` (seconds)

    **Read/Write**: Both

    **Conceptual role**: Sets the playback position in the root timeline (global coordinate)

    **Side effects**: 
    - Triggers frame updates across all child timegroups
    - Value is quantized to frame boundaries based on `fps`
    - Updates `currentTimeMs` automatically
    - If element has `id`, saves to localStorage on next seek

    **Constraints**: Should only be written on root timegroup. Child timegroups compute their time from parent state.

    **Example**: `timegroup.currentTime = 5.5` (sets to 5.5 seconds, quantized to nearest frame)

    For programmatic control that waits for frames to be ready, use the `seek()` method instead.

    For details on time propagation, see [Time Propagation](/docs/elements/timegroup/explanation/time-propagation).
  </PropertyDoc>
  <PropertyDoc
    name="currentTimeMs"
    domReadable
    domWritable
    htmlAttribute="currenttime"
    type="number"
    defaultValue="0"
  >
    **Type**: `number` (milliseconds)

    **Read/Write**: Both

    **Conceptual role**: Same as `currentTime` but in milliseconds

    **Side effects**: Same as `currentTime`

    **Relationship**: `currentTimeMs = currentTime × 1000`

    **Constraints**: Should only be written on root timegroup. Child timegroups compute their time from parent state.
  </PropertyDoc>
  <PropertyDoc
    name="ownCurrentTimeMs"
    domReadable
    type="number"
    defaultValue="0"
  >
    **Type**: `number` (milliseconds, read-only)

    **Conceptual role**: The current time within this element's local timeline (local coordinate)

    **Behavior**:
    - Sequence mode: Resets to 0 at this element's start time, progresses independently
    - Contain mode: Tracks parent's `ownCurrentTimeMs`
    - Fit mode: Tracks parent's `ownCurrentTimeMs` (identity mapping)

    **Use case**: Used in `addFrameTask` callbacks to get element-relative time

    **Key distinction**: `currentTimeMs` is global (root timeline), `ownCurrentTimeMs` is local (element timeline)

    For details on time coordinate systems, see [Time Propagation](/docs/elements/timegroup/explanation/time-propagation).
  </PropertyDoc>
  <PropertyDoc
    name="fps"
    domReadable
    domWritable
    htmlAttribute
    type="number"
    defaultValue="30"
  >
    **Type**: `number` (frames per second)

    **Read/Write**: Both

    **Conceptual role**: Controls frame quantization for time updates

    **Side effects**:
    - All time values snap to frame boundaries: `quantized_time = round(time × fps) / fps`
    - During rendering, `effectiveFps` uses render options FPS instead
    - Ensures consistent seek/render behavior and synchronization

    **Constraints**: Should be set on root timegroup to avoid quantization mismatches. Don't mix FPS values in nested timegroups.

    **Related property**: `effectiveFps` (read-only) - Returns actual FPS being used (render options FPS during rendering, otherwise this property)

    For details on frame quantization, see [Frame Quantization](/docs/elements/timegroup/explanation/frame-quantization).
  </PropertyDoc>
  <PropertyDoc
    name="effectiveFps"
    domReadable
    type="number"
  >
    **Type**: `number` (frames per second, read-only)

    **Conceptual role**: Returns the actual FPS being used for frame quantization

    **Behavior**:
    - During rendering: Returns FPS from render options
    - Otherwise: Returns the `fps` property value

    **Use case**: Internal - used by frame quantization logic to determine frame boundaries

    For details on frame quantization, see [Frame Quantization](/docs/elements/timegroup/explanation/frame-quantization).
  </PropertyDoc>
</PropertyDocList>

#### Visual Layout

<PropertyDocList>
  <PropertyDoc
    name="fit"
    domReadable
    domWritable
    htmlAttribute
    type="string"
    defaultValue="none"
  >
    Controls how the timegroup fits within its parent container visually. Can be `none`, `contain`, or `cover`.

    This property is used to control the visual scaling behavior when the timegroup's aspect ratio doesn't match its parent container.

    - `none`: No fitting applied, content maintains its natural size
    - `contain`: Scales content to fit within the parent while maintaining aspect ratio
    - `cover`: Scales content to fill the parent while maintaining aspect ratio

    **Note**: This is different from the temporal `mode` property. The `fit` property controls visual layout, while `mode` controls timeline behavior.
  </PropertyDoc>
</PropertyDocList>

### Methods

Programmatic control of `ef-timegroup` elements is available through JavaScript methods. You can get a reference to an element using `document.getElementById()` or a `ref` in frameworks like React.

<PropertyDocList>
<PropertyDoc
  name="addFrameTask(callback)"
  type="function"
>
  Registers a callback function to be executed on every frame update of the timegroup. This is useful for creating custom animations or synchronizing external elements with the timegroup's progress.

  -   **`callback`**: `(info: FrameTaskInfo) => void | Promise<void>`
      A function that will be called on each frame. It receives an `info` object with the following properties:
      -   `ownCurrentTimeMs`: `number` - The current time of this specific timegroup in milliseconds.
      -   `currentTimeMs`: `number` - The current time of the root timegroup in milliseconds.
      -   `durationMs`: `number` - The total duration of this timegroup in milliseconds.
      -   `percentComplete`: `number` - The progress of this timegroup, from 0 to 1.
      -   `element`: `EFTimegroup` - A reference to the timegroup element instance.
  -   **Returns**: `() => void`
      A cleanup function. Call this function to remove the registered callback.

  #### Example

  ```javascript
  const cleanup = timegroup.addFrameTask((info) => {
    console.log(`${(info.percentComplete * 100).toFixed(2)}%`);
  });
  ```
</PropertyDoc>
<PropertyDoc
  name="removeFrameTask(callback)"
  type="function"
>
  Removes a previously registered frame task callback.

  -   **`callback`**: `(info: FrameTaskInfo) => void | Promise<void>`
      The same callback function that was passed to `addFrameTask`.
</PropertyDoc>
<PropertyDoc
  name="seek(timeMs)"
  type="async function"
>
  **Type**: `async (timeMs: number) => Promise<void>`

  **Semantic responsibility**: Asynchronously seek to a specific time and wait for all visible frames to be ready

  **Parameters**:
  - `timeMs`: Time in milliseconds to seek to (will be quantized to frame boundary)

  **Behavior**:
  1. Sets `currentTime` (quantized)
  2. Waits for seek task completion
  3. Waits for all visible elements' frame tasks to complete
  4. Saves time to localStorage (if element has `id`)

  **Difference from currentTime setter**: 
  - `currentTime = 5` triggers async update but returns immediately
  - `await seek(5000)` waits until frames are ready for that time

  **Use cases**: 
  - Tests requiring frame-perfect assertions
  - Programmatic rendering
  - External timeline scrubbing

  **Example**:
  ```javascript
  await timegroup.seek(5000); // Seek to 5 seconds and wait for frames
  ```
</PropertyDoc>
<PropertyDoc
  name="isRootTimegroup"
  type="boolean"
>
  **Type**: `boolean` (read-only property)

  **Semantic responsibility**: Indicates whether this timegroup is the root (has no parent timegroups)

  **Returns**: `true` if no parent timegroup exists, `false` otherwise

  **Usage**: Internal logic uses this to determine time propagation rules
</PropertyDoc>
<PropertyDoc
  name="waitForMediaDurations()"
  type="async function"
>
  **Type**: `async () => Promise<void>`

  **Semantic responsibility**: Wait for all nested media elements (ef-video, ef-audio) to load their durations

  **Behavior**:
  1. Waits for all child media elements to connect
  2. Loads fragment index for each media element
  3. Flushes caches and updates timeline calculations

  **Use case**: Called automatically during initialization; can be called manually when dynamically adding media
</PropertyDoc>
<PropertyDoc
  name="renderAudio(fromMs, toMs)"
  type="async function"
>
  **Type**: `async (fromMs: number, toMs: number) => Promise<AudioBuffer>`

  **Semantic responsibility**: Render audio buffer for a time range (used by playback controller)

  **Parameters**:
  - `fromMs`: Start time in milliseconds
  - `toMs`: End time in milliseconds

  **Returns**: AudioBuffer containing mixed audio from all active media elements in range

  **Use case**: Called internally during live playback; can be used for audio preview/export
</PropertyDoc>
<PropertyDoc
  name="testPlayAudio(fromMs, toMs)"
  type="async function"
>
  **Type**: `async (fromMs: number, toMs: number) => Promise<void>`

  **Semantic responsibility**: Render and immediately play back audio (debugging utility)

  **Parameters**:
  - `fromMs`: Start time in milliseconds
  - `toMs`: End time in milliseconds

  **Returns**: Promise that resolves when playback completes

  **Example**:
  ```javascript
  await timegroup.testPlayAudio(0, 5000); // Play first 5 seconds
  ```
</PropertyDoc>
<PropertyDoc
  name="getMediaElements()"
  type="function"
>
  **Type**: `() => EFMedia[]`

  **Semantic responsibility**: Returns all media elements (ef-video, ef-audio) within this timegroup

  **Returns**: Array of media elements for audio rendering/playback

  **Use case**: Used internally by renderAudio() to collect media sources
</PropertyDoc>
</PropertyDocList>

### Storage Persistence

Timegroups with an `id` attribute automatically persist `currentTime` to `localStorage` using key `ef-timegroup-${id}`. Saves after each seek, restores on page load. Works only for root timegroups.

**Methods**: `loadTimeFromLocalStorage()`, `saveTimeToLocalStorage(time)`

See [How to Use LocalStorage Persistence](/docs/elements/timegroup/how-to/local-storage-persistence) for details.
