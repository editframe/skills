# Timegroup

The timegroup element is the fundamental organizing unit of an Editframe video. It is used to declare temporal relationships between elements in a video timeline.

The outer-most timegroup is the root timegroup. It is the parent of all other timegroups. The pixel dimensions of the root timegroup determine the aspect ratio of the video, and the computed duration of the root timegroup determines the video duration.

## Concept

The timegroup element implements **temporal composition**: a declarative system where:
- Parent timegroups define **composition rules** (how children relate in time)
- Child timegroups declare **durations** (fixed, computed, or derived)
- The system computes a **timeline tree** with precise start/end times for every element

**Core invariant**: Every timegroup has exactly one duration value at any moment, computed from either explicit specification (fixed mode) or child relationships (sequence/contain/fit modes).

## Temporal Relationships

Timegroups form a tree where each node's duration is determined by:

**Fixed mode**: `duration = explicit value` (base case)

**Sequence mode**: `duration = Σ(child.duration) - (n-1) × overlap` where n = number of children

**Contain mode**: `duration = max(child.duration)` across all children

**Fit mode**: `duration = parent.duration` (inheritance)

**Time coordinate systems**:
- `currentTime` = root timeline coordinate (global, only set on root)
- `ownCurrentTime` = element's local time coordinate (computed, resets per element)

**Propagation rules**:
- Sequence: Each child's `ownCurrentTime` progresses within its time-shifted window
- Contain: All children share the same `ownCurrentTime` as parent
- Fit: Child `ownCurrentTime` = parent `ownCurrentTime` (identity mapping)

## Invariants

1. **Root-only time writes**: Only the root timegroup's `currentTime` should be written. Child times are computed from parent state.

2. **Time quantization**: All time values snap to frame boundaries based on `fps`:
   ```
   quantized_time = round(time × fps) / fps
   ```
   During rendering, uses render options FPS; otherwise uses element's `fps` property (default: 30).

3. **Duration immutability per mode**: 
   - Fixed: Duration can be set via `duration` attribute
   - Sequence/Contain: Duration is computed from children (read-only)
   - Fit: Duration equals parent's duration (read-only)

4. **Own time coordinate system**: Each element has its own `ownCurrentTime` that:
   - Resets to 0 at element's start time
   - Progresses independently of siblings (in sequence mode)
   - Tracks parent time (in contain/fit modes)

5. **CSS variable scope**: `--ef-duration`, `--ef-progress`, `--ef-transition-*` are set per-element, not inherited via CSS cascade

6. **Overlap semantics**: In sequence mode, overlap creates **temporal intersection**, not visual crossfade. Visual transitions require explicit CSS animations.

7. **Fit vs mode distinction**: 
   - `fit` attribute controls **visual scaling** (CSS-like: none/contain/cover)
   - `mode` controls **temporal composition** (timeline behavior)

8. **Frame quantization consistency**: All time updates respect frame boundaries. Setting `currentTime = 1.234` with `fps=30` becomes `1.233...` (37th frame).

## Mental Model

```
Root Timegroup (mode=sequence, 10s total)
├─ Child A (mode=fixed, 5s)
│   └─ global window: [0s, 5s)
│   └─ ownCurrentTime: [0s, 5s)
├─ Child B (mode=fixed, 3s) 
│   └─ global window: [5s, 8s)
│   └─ ownCurrentTime: [0s, 3s)  ← RESETS to 0
└─ Child C (mode=fit)
    └─ global window: [0s, 10s)
    └─ ownCurrentTime: [0s, 10s)  ← INHERITS parent
```

When root's `currentTime = 6s`:
- Child A: inactive (6s is outside its [0s, 5s) global window)
- Child B: **active** with `ownCurrentTimeMs = 1000` (1s into its own 3s coordinate system)
- Child C: active with `ownCurrentTimeMs = 6000` (fit mode inherits full parent time)

**Key distinction**: 
- `currentTime` = position in root timeline (global coordinate)
- `ownCurrentTime` = position in element's local timeline (local coordinate, resets per element)

## Behavioral Consequences

From the composition rules above:

1. **Sequence items are serial in global time, independent in local time**: Changing Child A's duration shifts Child B's global window but Child B's `ownCurrentTime` still runs [0s, 3s)

2. **Contain items are parallel in both coordinates**: All children share same global and local time

3. **Fit items are duration-transparent**: They never affect parent duration calculation

4. **Overlap reduces global duration**: Each overlap subtracts from total: `duration = Σ(child) - (n-1) × overlap`

5. **CSS variables are local**: `--ef-progress = ownCurrentTime / duration` for each element

6. **Frame quantization is universal**: All time writes snap to frame grid, ensuring consistent seek/render behavior

## Common Misconceptions

### Writing currentTime to non-root timegroups
**Wrong**: `childTimegroup.currentTime = 5`

**Why**: Only root controls timeline; children compute their time from parent

**Right**: `rootTimegroup.currentTime = 5` (write to root only)

### Confusing currentTime vs ownCurrentTime
**Wrong**: Expecting Child B in sequence to have `currentTime = 1` when it's 1s into playback

**Why**: `currentTime` is global (6s), `ownCurrentTime` is local (1s)

**Right**: Read `ownCurrentTime` for element-relative time

### Confusing fit attribute vs fit mode
**Wrong**: `fit="contain"` expecting duration to change

**Why**: `fit` is visual scaling (CSS-like), `mode` is temporal behavior

**Right**: `mode="fit"` for duration inheritance, `fit="contain"` for visual scaling

### Expecting overlap to create visual crossfade
**Wrong**: `overlap="2s"` alone creates smooth visual transition

**Why**: Overlap only affects timing; CSS handles visuals

**Right**: Combine `overlap` with CSS animations using `--ef-transition-duration`

### Hard-coding animation timing
**Wrong**: `animation-delay: 3s` for fade-out

**Why**: Breaks when parent duration changes

**Right**: `animation-delay: var(--ef-transition-out-start)`

### Mixing FPS values in nested timegroups
**Wrong**: Setting different `fps` on parent (30) and children (60)

**Why**: Creates quantization mismatches and frame alignment issues

**Right**: Set `fps` once on root timegroup (propagates to all descendants)

### Using currentTime for programmatic control without seeking
**Wrong**: Setting `currentTime` and immediately reading frame state

**Why**: Frame updates are async; state may not be ready

**Right**: Use `await timegroup.seek(timeMs)` which waits for completion

## Frame Rate and Time Quantization

To ensure that animations and media playback are perfectly synchronized, `ef-timegroup` quantizes all time updates to a consistent frame grid. This is controlled by the `fps` (frames per second) attribute.

When you set the `currentTime` on a timegroup, the value is automatically rounded to the nearest frame boundary based on the configured `fps`. This process, known as time quantization, is crucial for preventing timing inconsistencies and producing predictable, smooth output, especially in a rendering environment.

## Methods

Programmatic control of `ef-timegroup` elements is available through JavaScript methods. You can get a reference to an element using `document.getElementById()` or a `ref` in frameworks like React.

<PropertyDocList>
<PropertyDoc
  name="addFrameTask(callback)"
  type="function"
>
  Registers a callback function to be executed on every frame update of the timegroup. This is useful for creating custom animations or synchronizing external elements with the timegroup's progress.

  -   **`callback`**: `(info: FrameTaskInfo) => void | Promise<void>`
      A function that will be called on each frame. It receives an `info` object with the following properties:
      -   `ownCurrentTimeMs`: `number` - The current time of this specific timegroup in milliseconds.
      -   `currentTimeMs`: `number` - The current time of the root timegroup in milliseconds.
      -   `durationMs`: `number` - The total duration of this timegroup in milliseconds.
      -   `percentComplete`: `number` - The progress of this timegroup, from 0 to 1.
      -   `element`: `EFTimegroup` - A reference to the timegroup element instance.
  -   **Returns**: `() => void`
      A cleanup function. Call this function to remove the registered callback.

  #### Example

  Here is an example of using `addFrameTask` to display the progress.

  ```html
  <ef-timegroup
    id="frame-task-example"
    mode="fixed"
    duration="5s"
    class="w-[500px] h-[300px] bg-slate-500 text-white flex flex-col items-center justify-center"
  >
    <p class="text-4xl">Frame Task Example</p>
    <p id="frame-task-progress" class="text-2xl font-mono"></p>
  </ef-timegroup>
  ```

  ```javascript
  const timegroup = document.getElementById('frame-task-example');
  const progressEl = document.getElementById('frame-task-progress');

  const cleanup = timegroup.addFrameTask((info) => {
    progressEl.textContent = `${(info.percentComplete * 100).toFixed(2)}%`;
  });

  // You would call cleanup() when the component unmounts or is no longer needed.
  ```

<PropertyDoc
  name="seek(timeMs)"
  type="async function"
>
  **Type**: `async (timeMs: number) => Promise<void>`

  **Semantic responsibility**: Asynchronously seek to a specific time and wait for all visible frames to be ready

  **Parameters**:
  - `timeMs`: Time in milliseconds to seek to (will be quantized to frame boundary)

  **Behavior**:
  1. Sets `currentTime` (quantized)
  2. Waits for seek task completion
  3. Waits for all visible elements' frame tasks to complete
  4. Saves time to localStorage (if element has `id`)

  **Difference from currentTime setter**: 
  - `currentTime = 5` triggers async update but returns immediately
  - `await seek(5000)` waits until frames are ready for that time

  **Use cases**: 
  - Tests requiring frame-perfect assertions
  - Programmatic rendering
  - External timeline scrubbing

  **Example**:
  ```javascript
  await timegroup.seek(5000); // Seek to 5 seconds and wait for frames
  ```

## Storage Persistence

Timegroups with an `id` attribute automatically persist `currentTime` to `localStorage`:

**Storage key**: `ef-timegroup-${id}`

**Automatic behavior**:
- Saves `currentTime` after each seek
- Restores `currentTime` on page load (if key exists)
- Works only for root timegroups with `id` attribute

**Example**:
```html
<ef-timegroup id="my-video" mode="fixed" duration="60s">
  <!-- Will remember playback position across page reloads -->
</ef-timegroup>
```

**Methods**:
- `loadTimeFromLocalStorage()` - Manual load (called automatically on connect)
- `saveTimeToLocalStorage(time)` - Manual save (called automatically after seek)

## Advanced Composition Patterns

### Nested Sequences with Transitions

```html
<ef-timegroup mode="sequence" overlap="1s">
  <ef-timegroup mode="sequence">
    <ef-timegroup mode="fixed" duration="3s">Sub A1</ef-timegroup>
    <ef-timegroup mode="fixed" duration="2s">Sub A2</ef-timegroup>
  </ef-timegroup>
  <ef-timegroup mode="fixed" duration="4s">B</ef-timegroup>
</ef-timegroup>
```

**Timing analysis**:
- Inner sequence: 5s total (3s + 2s)
- Outer sequence: 8s total (5s + 4s - 1s overlap)
- B starts at 4s global time (5s - 1s overlap), runs until 8s

### Background Track Pattern (Fit Mode)

```html
<ef-timegroup mode="contain">
  <ef-timegroup mode="sequence" class="foreground">
    <ef-timegroup mode="fixed" duration="3s">Intro</ef-timegroup>
    <ef-timegroup mode="fixed" duration="10s">Main</ef-timegroup>
    <ef-timegroup mode="fixed" duration="2s">Outro</ef-timegroup>
  </ef-timegroup>
  <ef-timegroup mode="fit" class="background">
    <!-- Automatically spans full 15s duration -->
  </ef-timegroup>
</ef-timegroup>
```

**Consequence**: Background duration = max(foreground) = 15s (fit mode inherits)
